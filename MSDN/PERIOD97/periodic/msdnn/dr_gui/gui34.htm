Many, many, many thanks to Dr. GUI’s chief specialist Saji Abraham, who is as of this column a certified miracle worker. (Six more and we give him a feast day!)

Dr. GUI Gets Wired (or Is that Weird?) on Java

About this time last year, Dr. GUI got Webbed. This year, Dr. GUI’s been drinking Java™ from a fire hose—he just got back from Sun Microsystem’s annual JavaOne conference for Java programmers and he’s still wired. It was a busy week—so busy that Dr. GUI didn’t even have any time for sightseeing or fun in his favorite city, San Francisco. His diligence to his duties paid off, however, and Dr. GUI has come back with a great prognosis for the future of Java in our industry. But he also diagnosed a severe pathological condition that will take all of our efforts to cure. More on that later.

The Good News

You’ll recall that the good doctor has long been a fan of Java. Dr. GUI likes the elegance of the language and the fact that his code will run anywhere there’s a Java Virtual Machine (VM). (It might run a little slowly and it might not take full advantage of the platform, but it’ll run.) But the good doctor’s also been frustrated about the limitations of Java—that the API hasn’t really delivered the functionality needed be to taken at all seriously as a platform. (Thank goodness for COM, which allows him to work around those limitations, at least on Win32®.) Dr. GUI does as much coding as he can in Java, but given limitations of speed and functionality, he can’t do all of his surgery in Java.

The good news from JavaOne is that the capabilities of Java are rapidly expanding, starting with the soon-to-be-available-most-everywhere version 1.1. Although the JavaSoft division of Sun Microsystems has not done much to enhance Abstract Window Toolkit (AWT), which is widely regarded as the weakest part of Java, they have been working hard on a number of very significant and important changes. In fact, they’ve added over 20 packages, containing well over 200 classes, to the Java Development Kit (JDK). The JDK is getting so large, in fact, that JavaSoft is planning on offering trimmed-down Java standards called “Personal Java” and “Embedded Java” so that parts of the Java run time can be implemented without huge memory requirements. (For instance, “Personal Java” is supposed to take “only” 2 megabytes of memory.) Let it be noted that the good doctor is concerned about these new standards (not to mention “Enterprise Java”) fragmenting Java. After all, is it really “Write ONCE, Run Anywhere™” if it won’t run on my “Personal Java” set-top box?

New API from JavaSoft

These new classes address many crucial issues for Java developers, such as internationalization, components (via the JavaBeans™ component model), remote method invocation, and more flexible security, including signed applets using technology somewhat similar to Microsoft® Authenticode™. JavaBeans in particular is very cool. Sun did a wonderful demo of Beans, including some very cool visual development tools and Beans that had been wrapped as an ActiveX™ control being used in all sorts of ActiveX containers—Microsoft Visual Basic®, Microsoft Excel, and even Microsoft Internet Explorer 3.0. It’s interesting that wrapping a Bean as an ActiveX control is the only way to use it in a Web browser at this point, since other browsers don’t yet support Beans. (Yes, Dr. GUI knows that it’s possible to host a Bean that is also an Applet in any Java-enabled browser, but that’s hosting an applet, not just any Bean.) Although this was an impressive demonstration of the power of ActiveX, it’s even more impressive to think of the power you’ll get when ActiveX controls and JavaBeans are fully integrated in the next version of the Microsoft Win32 Virtual Machine for Java, currently in beta. (Read on.)

JavaSoft promised future advances in their 2- and 3-D imaging models, security, an enterprise framework, and more for future releases. Given JavaSoft’s success with the 1.1 release, Dr. GUI has high hopes that he won’t be waiting long for more great Java enhancements. And JavaSoft seems to be doing a good job hitting functional areas that most developers need.

Putting Some Meat on AWT

There will be two approaches to beefing up AWT, Java’s windowing library. Sun announced the Java Foundation Classes (JFC), which sounded like a rewrite of Netscape’s rather lackluster Internet Foundation Classes (IFC). If you’re developing in IFC, don’t worry: there will be a migration kit. (What fun!) Sun announced that they had engineers from Netscape, IBM, and Sun working on JFC, and that they’d be out for beta by late summer. But frankly, I wasn’t impressed with either the functionality they described (limited and inflexible compared with Microsoft’s Application Foundation Classes) nor the schedule (I want it yesterday!). But the good doctor did like the pluggable look-and-feel features of JFC.

What’s New from Microsoft

Dr. GUI was, however, most impressed with Microsoft’s Application Foundation Classes (AFC) for Java. In fact, he saw a demo at Microsoft where a Microsoft developer asked, “When are you going to port this stuff back to C++?” When was the last time you saw C++ developers drooling over a Java API library? It’s that good.

There’s a beta version of AFC available now (check it out at http://www.microsoft.com/java/pre-sdk/default.htm), and the full documentation is available as well (same place, click “Packages and Classes” in the left pane). But don’t take Dr. GUI’s word for it—check Simson Garfinkel’s article in Packet (see http://www.packet.com/packet/garfinkel/97/12/index2a.htm). Dr. GUI knows that when you compare the two libraries for yourself you’ll agree that AFC rocks. And it’s 100% Java code—it’ll run anyplace there’s a Java VM.

In addition, the next version of the Microsoft Win32 Virtual Machine for Java will support full two-way interaction between ActiveX controls and Java Beans, including visual controls. That means you can use a Java Bean anywhere you can use an ActiveX control (such as Visual Basic, Microsoft Visual C++®, Microsoft Access, Microsoft Excel, Microsoft Word, Delphi, and so on) and that you can use an ActiveX control written in any language anywhere you can use a Java component. (Check out a Java version of Internet Explorer at http://www.packet.com/packet/garfinkel/97/12/code.html. Only 150 lines of Java with AFC—and the help of the IE Web browser ActiveX control.) The AFC library is based on standard Java/AWT Container/Component/LayoutManager architecture, and ActiveX controls appear in the Java world as Components. That means you can even build a tree or list control populated with ActiveX controls—even Visual Basic controls or Web browsers! Note that this VM support is completely separate from AFC—AFC is 100% cross-platform. This VM is also in beta now (you’ll find it at http://www.microsoft.com/java/). You can start developing for both it and AFC today.

Finally, check out AFC Enterprise Libraries (http://www.microsoft.com/java/). Some of the features, such as transaction processing services, are in beta now. Other features, such as advanced database access, directory services, enterprise management services, and object interoperability using DCOM, will be in beta soon and will be shipping sometime in the second half of this year. And if you’re developing Java apps now, don’t miss J/Advantage (also at http://www.microsoft.com/java/).

Java Is Real!

The key message is that, more than ever before, Java is a language and run-time system in which you can get real work done—all while enhancing the investment in the programs you write because they’ll run in many different environments without requiring recompilation. (Dr. GUI never has liked porting code except for MFC apps, which almost port themselves.) If you haven’t paid attention to Java before, Dr. GUI recommends doing so now—there are now many applications for which Java is the smartest choice for you and your customers. But make no mistake about it: Java will always be slower than good C++ code (the Sun attitude of “it’s more important to have a secure system than a fast system” ensures this). And cross-platform programs truly are “lowest common denominator” programs, no matter how much the bottom is raised—they will never be able to take advantage of all of the great features of every platform. Despite these limitations, there are many applications for which Java is the right choice, so Java is always worth considering and choosing when it makes good sense. And with the rapid advances in Java programming libraries and tools, it makes sense more and more often.

The Java Religion

Java has, for better or worse, transcended being a mere technology and has become a religion. The leading school of thought in this religion is the “100% Pure Java” movement (see http://java.sun.com/100percent/), which asks developers to “take the pledge” to write only 100% Pure Java. (They even give out “take the pledge” buttons you can wear.) This religion preaches salvation through its mantra, “Write Once, Run Anywhere.”

The Key Question

The whole time he was at the conference, Dr. GUI was dying for someone to ask the key pertinent question: “I’ve got customers today who need functionality that’s not provided in ‘100% Pure Java.’ They want the solutions now—I can’t wait for the next release. If I don’t serve them, my competitors will. What does Sun say I should do?”

Dr. GUI really wanted to hear the answer to that one. It would have spoken volumes about the speaker’s view of developers and customers. (The good doctor would have asked himself, but he considers it too rude at another company’s conference.)

Dr. GUI on Programming Religion

Dr. GUI loves Java, the technology; but he rejects Java, the religion. If, for programming’s sake, Dr. GUI “needs” to “get religion,” then the good doctor’s first commandment will be this: “Serve thy customer to the best of thy ability.”

If customers need 100% Pure Java, great. If they need “99 44/100% Pure Ivory Java,” that’s fine, too. Dr. GUI doesn’t care what color the computer is as long as it runs code. Don’t you think that what developers and customers really need is “100% useful Java,” not a religious experience? Wouldn’t it be better to work together to make Java 100% useful?

As he absorbed the religious fervor surrounding the conference, it occurred to Dr. GUI that with just one simple letter change, he could become Fr. GUI (hey, he’s been a toaster—what’s a religious conversion compared to that?). He quickly rejected that notion, preferring to remain among the laity. Dr. GUI nonetheless mused that, if he were to write a closing prayer for the conference, he would take as his theme, “Can’t we all just get along?”
