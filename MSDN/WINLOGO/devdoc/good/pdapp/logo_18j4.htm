Provide Consistent, Up-to-date Windows Support

[This is preliminary documentation and subject to change.]

Providing consistent, up-to-date support for Windows standards and technologies makes your application easier to use and administer, and helps to avoid incompatibility problems.

Designed for Microsoft® Windows® Logo recommendations and best practices that relate to this goal are listed below. The lists are just summaries. To comply with the recommendations and best practices, you must follow all stipulations of the reference text to which the lists refer.

General

  • Support additional standard system settings. (Recommended)
  • Use standard system settings for multi-monitor support. (Best Practice)
  • Support Windows Sockets 2.0. (Recommended)
  • Detect and handle slow or dead network links. (Best Practice)

File System

  • Be file-system independent. (Best Practice)
  • Hide program files and directories that users do not need to see. (Best Practice)

Operating System Migration

  • Migration DLLs work silently. (Recommended)
  • Avoid operating system differences. (Best Practice)

Accessibility and User Interface

  • Do not rely on sound alone. (Recommended)
  • Do not rely on color alone. (Recommended)
  • Let other software identify and manipulate screen elements. (Recommended)
  • Support Active Accessibility. (Best Practice)
  • Do not hard-code font sizes smaller than 10 points. (Best Practice)
  • Allow users to customize UI timings. (Best Practice)
  • Support system font and resolution changes. (Best Practice)
  • Allow users to set font names and sizes. (Best Practice)
  • Handle changes in pointer or keyboard focus. (Best Practice)
  • Test using multiple displays. (Best Practice)

Internet

  • Support automatic download of ActiveX controls. (Best Practice)
  • Register an Internet update site when installing. (Best Practice)
  • Register a product information site when installing. (Best Practice)
  • Support ModuleUsage registration of DLLs. (Best Practice)
  • Expose Help documentation as HTML. (Recommended)
  • Support saving to an FTP site. (Best Practice)
  • Generate browsable output. (Best Practice)

OLE / COM

  • Make object servers ActiveX document servers. (Recommended)
  • Make containers ActiveX document containers. (Recommended)
  • Close object servers automatically after object delivery. (Recommended)
  • Support ActiveX Accessibility in containers. (Recommended)
  • Support modifier keys in containers. (Recommended)
  • Provide linking and in-place activation in containers. (Recommended)
  • Support the structured storage compound file format in containers. (Recommended)
  • Use component categories. (Best Practice)
  • Use the system property set implementation. (Best Practice)
  • Provide a type library for public automation interfaces. (Best Practice)

COM+

  • Support Microsoft Transaction Server. (Best Practice)
  • Expose all new object servers as COM+ objects. (Best Practice)
  • Support the COM+ versioning and configuration model. (Best Practice)
  • Use COM+ calling conventions. (Best Practice)
  • Support COM+ events rather than custom event programming. (Best Practice)
  • Expose standard COM+ interfaces. (Best Practice)
  • Take advantage of COM+ run-time interfaces. (Best Practice)
  • Take advantage of COM+ run-time DLL functions. (Best Practice)
  • Do not write InprocServer32 entries for components directly to HKCR. (Best Practice)
  • Support COM+ declarative attributes. (Best Practice)
  • Use COM+ automatic transaction support, persistence, and data binding as necessary. (Best Practice)
  • Implement system monitoring or call monitoring and management using interception. (Best Practice)
  • Use the COM+ persistent event binding system. (Best Practice)
  • Use the COM+ importer, emitter, portable executable format, and run-time system. (Best Practice)

Telephony

  • Use TAPI 2.1 remote service provider for client-server telephony. (Recommended)
  • Use TAPI 2.1 client-management functions for client-server telephony. (Recommended)
  • Use TAPI 2.1 for telephony as soon as it becomes available. (Best Practice)
  • Use the dialing properties current location settings in telephony-enabled applications. (Best Practice)
  • Use the canonical/international form for the number dialed in telephony-enabled applications. (Best Practice)
  • Provide separate fields with defaults for phone number entry. (Best Practice)
  • Handle the LINE_CREATE message properly in telephony-centric applications. (Best Practice)
  • Handle _REINIT messages properly in telephony-centric applications. (Best Practice)

OnNow / ACPI Support

  • Eliminate even minor memory leaks. (Recommended)
  • Process Sleep or Query messages silently. (Recommended)
  • Pause play and sound on receiving a Sleep message in a game. (Recommended)
  • Call SetThreadExecutionState in event-handling applications. (Best Practice)
  • Call SetThreadExecutionState in presentation applications. (Best Practice)
  • Curtail background activities when running on batteries. (Best Practice)
  • Take no action on receiving a …RESUMEAUTOMATIC message. (Best Practice)

Directory Services

  • Use a product-specific DS object in the configuration/services container. (Best Practice)
  • Use RpcNsBindingLookupBegin in RPC clients to find RPC server objects. (Recommended)
  • Use WSALookupServiceBegin in clients to find Windows Sockets server objects. (Recommended)
  • Locate COM objects in the Class Store transparently in COM and DCOM clients. (Recommended)
  • Publish only machine-specific parameters in the registry. (Recommended)
