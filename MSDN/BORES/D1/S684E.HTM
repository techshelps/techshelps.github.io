Network Protocol Issues

SQL Executive requires a Net-Library that allows a trusted connection. The only Net-Libraries that support a trusted connection are named pipes and multiprotocol.

Additionally, if you have ever had a parallel installation of SQL Server with both version 4.21 and version 6.0 running on the same machine and SQL Server version 6.0 or version 6.5 is set up to listen on an alternate named pipe, you must change the registry entry for the ServerHost value for SQL Executive so it connects to the correct pipe.

Caution Incorrect changes or deletions made to the Registry can damage Windows NT and prevent the system from running normally.

Changes to the ServerHost value in the Registry can be found in:

HKEY_LOCAL_MACHINE on Local Machine
   \SOFTWARE
      \MICROSOFT
        \MSSQLServer
           \SQL Executive
              \ServerHost

For example, if you have set up SQL Server 6.5 to listen on \\.\Pipe\Mssql\Query, you would enter this for the ServerHost value. Stop and restart SQL Server for the changes, particularly the network protocol, to take effect.

If SQL Server is set up to listen on multiprotocol and not on named pipes, the SQL client configuration utility must also have multiprotocol as the default Net-Library. In addition, the Advanced tab of the SQL Client Configuration Utility must include the Windows NT computer name in the Server box and MultiProtocol in the DLL Name box. Click Add/Modify to add this advanced configuration if it is not already present. You must also add the server name to the ServerHost key in the Registry as listed above.
