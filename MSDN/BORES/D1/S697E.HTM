Read Consistency

SQL Server provides both statement-level and transaction-level read consistency. This produces repeatable reads, an important feature for long running transactions with multiple queries that need to see the same image of the data throughout.

SQL Server allows applications to achieve the best balance between read consistency and concurrency by supporting configurable transaction isolation levels. Transaction isolation levels determine how transactions interact with one another, as described in the following table.

Isolation level

Description

READ UNCOMMITTED

Directs SQL Server not to issue shared locks and does not honor exclusive locks during reads. At this level, applications may read values within uncommitted transactions that might get rolled back later ("dirty reads"), and therefore should be used only by transactions that are insensitive to this behavior.

READ COMMITTED

Directs SQL Server to use shared locks while reading data (the default behavior). At this level, transactions cannot experience "dirty reads."

REPEATABLE READ |
SERIALIZABLE

Guarantees that data read repeatedly within a transaction will retain the same values and that "dirty reads," nonrepeatable reads, and "phantom" values (values that are rolled back after being read) cannot occur. READ REPEATABLE and SERIALIZABLE are interchangeable.


You can optimize this behavior through configurable transaction isolation levels, which can be set globally for a session or individually for each SELECT statement.
