Copying Files with cpy2root.pdf

CPY2ROOT.PDF copies all files from root of the source directory to the root of c: on the client from which the PCM job is run. CPY2ROOT.PDF copies SU.INI and other files to the client before running a remote SMS installation job, which expects the files to be there.

Installing a Primary Site Using Package Command Manager

Note The computer on which you want to install an SMS site must currently be a client, and it must meet the requirements for primary site servers, as specified in SMS Getting Started.

Modify the initialization file, SU.INI, to include all the data needed to install the site.

Copy SU.INI to the client's c:\ root manually, or use a Run Command on Workstation job to copy it there. (The initialization file must be on c:\ and must be named SU.INI).

Modify and copy updproxy.bat to c:\ if you want the proxies to be copied automatically to their administrative installations.

Modify address.ini to include any addresses you want created, and place address.ini at the location you specified in SU.INI.

You can import CPY2ROOT.PDF to create a workstation package whose source directory contains su.ini and updproxy.bat.

The program to install SMS remotely (BVTDLG.PCD) carries out the following tasks if their keys are specified in su.ini:

  • Installs SMS
  • Copies proxy applications to an administrative installation of an application
  • Sets licensing, tracing file sizes, SMS Administrator defaults, and some site properties
  • Adds domains, package servers, and addresses
  • Attaches to a parent
  • Initiates the installation of secondary sites
  • Creates packages, queries, machine groups, program groups, site groups, and alerts
  • Creates jobs to compress and send packages (no instruction files)

Ensure that the client's site server has the MSTEST files. (COPYVT.PDF automatically copies the appropriate files.)

Import sites.pdf to create a workstation package whose source directory contains the sms source files and bvtdlg.pcd. Choose the command line Create new site. The smssetup directory must be exactly one level down from the root of the source directory; for example, c:\sms12\smssetup, with bvtdlg.pcd placed in sms12, and c:\sms12 specified as the source directory.

If you also include upgrade.pcd in the source directory, you can use the same package for installing new sites and for upgrading existing sites. Because the source directory is large, using the same package saves much disk space.

If you are sending the package from site A to install a site on a client of A, the procedure is the same as for any Run Command on Workstation job. When you are sending the package from site A to upgrade site B, which is lower in the hierarchy, proceed as follows:

Create a Run Command on Workstation job to send the package to site B (which need not be a default distribution server of site A).

In the job, under Put on Specified Distribution Servers, select Default Servers. The job will appear as a workstation package on the default distribution servers of site B (not site A) and can be run from the client of site B.

Upgrading a Primary Site Using Package Command Manager

Note Neither su.ini nor updproxy.bat is needed to upgrade a site using the following method.

Ensure that the site server to be upgraded has the mstest files. Because the client running the upgrade package is the site server itself, you must always send the files corresponding to the platform being upgraded; for example, if you are upgrading an x86 site, send the x86 mstest files. (COPYVT.PDF automatically copies the appropriate files.)

Import sites.pdf to create a workstation package whose source directory contains the sms source files and upgrade.pcd.

Choose the command line Upgrade existing site. The smssetup directory must be exactly one level down from the root of the source directory; for example, c:\sms12\smssetup, with upgrade.pcd placed in sms12, and c:\sms12 specified as the source directory.

If you also include bvtdlg.pcd in the source directory, you can use the same package for installing new sites and for upgrading existing sites. Because the source directory is large, using the same package saves much disk space.

When you are sending the package from site A to upgrade site A, the procedure is the same as for any workstation job . When you are sending the package from site A to upgrade site B, which is lower in the hierarchy, proceed as follows:

Create a Run Command on Workstation job to send the package to site B (which need not be a default distribution server of site A).

In the job, under Put on Specified Distribution Servers, choose Default Servers. The job will appear as a workstation package on the default distribution servers of site B (not site A) and can be run from site B.

If disk space is a concern, do not choose Default Servers. Instead, create a machine group with the minimum number of distribution servers needed to reach all sites.

Initialization File

This topic lists SMS initialization file parameters commonly used in SU.INI. This topic contains one subtopic for each section in SU.INI. The SU.INI file is used in the SMS test team's build verification and test (BVT), the BackOffice SMS install and verification script, and in other SMS testing. su.ini is commonly found in the root of the C: or D: drive.

Important Although the scripts check some of the values in SU.INI, you must ensure that SU.INI contains valid values.

SKIP means use the default. In some cases, such as SQL Server information, you might have to specify values.

To add a comment to SU.INI, start a line with an apostrophe (') or semicolon (;). You cannot put a comment on the same line as a non-comment.
