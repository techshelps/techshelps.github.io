Automatic Logon

Figure 5.3 Graphical representation of automatic logon.

When a user activates an SNA application on the client computer, it works with the SNA Server subdomain to perform resource location to determine which SNA Server computer and connection to use to open the LU session.

The SNA application on the client computer provides a replacement keyword as a placeholder for security information during session initiation. The format depends on the type of LU session established. For example, the string "MS$SAME" is used for APPC or C-PIC applications.

The SNA Server (or node) detects the replacement keyword and determines the Windows NT user name under which the client is logged on. This step limits automatic logon support to users running native client-server sessions, since only these sessions support Windows NT authentication. For example, TN3270 is not supported because the TN server cannot determine the client's Windows NT user name.

The node uses resource location to locate the host account cache ¾ either a backup up SNADATABASE service running on the same machine as SNA Server

or the master host account cache in the Windows NT domain running SNA Server. The node sends a lookup message to the host account cache. The request includes the Windows NT user name and host security domain name. It requests the corresponding host account name and password.

The SNADATABASE service checks that the Windows NT account exists in the host account database, and that the account is still a member of users group in the Windows NT domain representing the host security domain. If any of these checks fails, the user record is purged from the host account cache. If all checks pass, the SNADATABASE service returns a reply containing the host account name and password to SNA Server, using encrypted RPC. SNA Server substitutes the host account name and password into the SNA data stream and the host computer sees a regular session start up request containing the host account information.
