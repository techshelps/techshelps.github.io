Terminology of OmniDocs Engine

Commonly used key terms related to OmniDocs DMS Engine are described below to provide good understanding of the systemSystem.

 

 

DMS

 

Abbreviation for Document Management System. A DMS provides filing documents and other operations on documents. As in case of OmniDocs, a DMS may also introduce new classifying entities like folders Folders to facilitate easier document Document managementManagement.

 

Cabinet

 

Database Cabinet is the basic entity Entity in the OmniDocs Document Management System. OmniDocs Engine has 3 types of Cabinets. Viz. Document Cabinet, Image Cabinet, Both type Cabinet. A Cabinet refers to a database.

 

  • Document Cabinet - Document Cabinet refers to the database, which maintains the Document Management System information. All Users, folder Folder and document Document information resides in this Cabinet.

  • Image Server Cabinet - This Cabinet refers to the database where Volume and document Document storage Storage information is stored.

  • Both Type Cabinet - Currently both type Cabinet is being used . It encapsulates the features of both the Document Cabinet as well as Image Cabinet.

 

Cabinet can have properties Properties such as Security Level and Versioning.

 

Security level A Cabinet can be assigned security Security level Level during creationCreation. Security level Level identifies the level of Rights validation process.

 

0 -

No securitySecurity. No Rights validation is done for any User.

1 -

Cabinet Level Security i.e.that is, a User who has Rights on Cabinet has similar Rights on all   objects.

2 -

Object Level Security. Rights validation on each object Object is done. User can be granted Rights on documentsDocuments, foldersFolders, etc and Rights validation is done to ensure that User has required Rights before he can perform any operation on the objectObject.

Rights

One of the main Main features Features of OmniDocs Engine is that User can share documents, folder, Data class Data Class and Annotations with other Users of the Group. User can also customize the sharing of Objects by assigning Access Rights on the objectObject. Depending upon the Rights, oObject will be available to the authentic Users and can be hidden from others. This provides security Security to the objects Objects created by by the User and at the same time the data Data flow Flow is also not interrupted.

 

Rights Calculation process

 

The Right string is a six-bit string where each bit corresponds to the Right of performing some operation on the objectObject. The meaning of each bit is as given below  (from left to right).

Rights for any object Object are stored in the form of an ACL. It is stored as a comma separated list of Rights for individual Users or Groups, which in turn is in the format  - U/G{Index}# rights string. For e.g. if ACL for a document is U23#100111,G3#000000 it means that User with User Index 23 has Rights 100111 and Group with Group Index 3 has Rights  000000 on the document.

 

Algorithm for calculation of Rights on an object

 

Step 1 - Check the explicit Rights of the User on the object. If no explicit Rights are assigned, then proceed to step 2.

Step 2 - Get the explicit Rights on that object ,assigned to all the Groups to which the User belongs. ORing of Rights of all such Groups gives the final Rights. However, if no Rights are assigned to any such Group, proceed to step 3.

Step 3 - Repeat steps 1 & 2 for the parent of this object. If no Rights are returned for the parent also, then continue the process with the subsequent parent objects.

 

This sequence is to be repeated till the Rights are obtained.

Image Server

 

Image Server handles the storage Storage and retrieval Retrieval of documents in WAN environment. It stores documents using File SystemSystem and maintains information about these documents in database. Image server Server has two essential components viz.

 

         Image Cabinet This is the database component of Image Server. It maintains information of all documents stored in Image Server. This information includes location and size of document and other attributes relevant to mapping to actual document on File SystemSystem.

         Storage Management Server (SMS) This component provides the actual physical storage for the documents.

 

Image Server provides a client interface for communicating with Image Cabinet and SMS.

 

Image server consists of 3 basic building blocks. These building blocks are

         Site

         Volume

         VolBlock

 

A Site is a collection of Volumes and a Volume in turn is a collection of VolBlocks.

 

Basic terms used in Image Server are:

 

Image Cabinet

 

This is the database component of Image Server. It maintains information of all documents stored in Image Server. This information includes location, size of document and other attributes relevant to mapping of document on File System.

 

SMS 

SMS is the Storage Management Server, which listens for requests from Image server Server client and consequently performs document addition or retrieval or any other operation on request.

 

Site

A Site represents a SMS. A Site is identified by two attributes viz.namely, SMS IP, SMS Port, the IP of the server Server where SMS is running and the port Port on which it is listening. A Site serves as a repository for volumeVolumes each consisting of multiple volblocks. Site name serves as a logical name for identifying siteSites.

 

Volume

A Volume is a logical unit, which can have multiple VolBlocks. Image storage at a particular Site is divided into logical storage units called Image Volumes. An Image Volume is used to Group several Image Volume Blocks where each Image Volume Block corresponds to a document data file, which is a Group of one or more document files concatenated. There is no limit on the physical storage occupied by an Image Volume. Every volumeVolume has a default volumeVolume path, which is a label on SMS, which corresponds to a physical location on SMS.

 

Volume Block

 

Volume Blocks provide the actual physical storage for the documents stored in an Image Volume. Image Server for an Image Volume Block allocates no actual physical space at the time of creation; the space is allocated as and when required. The size of Image Volume Block is specified at the time of volumeVolume creationCreation. This size refers to the maximum size of the document data file that will be created corresponding to each VolBlock in this volumeVolume. Documents are added serially in the document data file until the maximum size is reached. A new VolBlock is created for the next document to be added after the maximum size is reached. This VolBlock size is flexible and can be modified after volumeVolume creation also.

If a document larger in size than the max VolBlock size is added it is dumped completely in the current volumeVolume block and a new one is created for next document. The concatenated string of Site Id, Volume ID and VolBlock ID in base 36 notations and prefixed by PN forms the name of a document data file in which data of the volblock is stored. This PN file provides the actual physical storage of documents. These files contain actual document together with Image Cabinet recovery information as headers.

 

Label

 

A Label basically represents the physical path where the data files corresponding to volblocks will be stored. Storage on any siteSite is broken among multiple labels, for better performance.

 

VolBlockPath

 

It represents the path of the actual data file on the SMS, corresponding to a volblock. It is stored in the format SMS: LabelName, here LabelName is the Label in which the volblock is added.

 

Document

 

A document is the basic unit of Image Server. Documents are stored in data files created corresponding to VolBlocks. Document operations include

 

Document Addition Adding a document to the Image   Server.

 

Document Retrieval By specifying the document indexIndex and the volumeVolume indexIndex, documents can be completely or partially retrieved from the Image Server. Document can be retrieved page wise also.

Document Deletion  Deletion of document from image server is two-step processes.

 

In first step document is only marked for deletion. This facilitates recovery of document after deletion i.e.that is, document can be salvaged.  The documents marked for deletion are deleted permanently when compact Compact Volume operation is performed. After compaction, documents cannot be recovered.

 

Document Replication Document can be replicated over multiple siteSites. This serves dual purpose of backup and fast access.

 

Volume Replication

 

A volumeVolume can be replicated over multiple siteSites. This feature provides two advantages.

         Instant Backup- Replication generates immediate backup. In case of loss of data on one siteSite it can be easily recovered from another siteSite.

         Faster retrieval - The idea is to replicate document data to all concerned siteSites so that a client can access data from its local cluster thereby reducing the need for Remote Access.

 

Volume Compaction

Volume Compaction is release of unused disk space occupied by the Volume. Delete Document functionality of the Image Server API marks a document for deletion and does notnt actually releases the physical space .The task of revoking the unused space is done during Volume Compaction.   

 

Volume Recovery

Image Server provides a mechanism for recovery of Image Cabinet data. It is used when the data file corresponding to a volblock exists but its information from the Image Cabinet is lost. It recovers all the information from the header information, which is stored in the data file. User has to specify the full path of data file on SMS in the format SMS:LabelName\FileName.

 

Home Site

Home siteSite for any volumeVolume, is the siteSite where that volumeVolume was originally added. If a volumeVolume is replicated on some other siteSite(s) then the siteSite in which it is originally added is the Home Site and the siteSite(s) where the volumeVolume is replicated are the Replicate Site(s).

 

Preferred Site Vs Remote Site

 

Image Server client while requesting for data retrieval, specifies the indexIndex of document it wants to retrieve, and also the siteSite indexIndex from where to fetch the document. This siteSite is called the Preferred Site for this client and all other siteSites in the systemSystem are Remote Sites. The Image Server checks whether the document is present on Preferred Site. If it is, then document is delivered from local SMS else a Remote Access is made to the siteSite, where the document is present. This approach results in faster access because User can specify the local SMS as the preferred siteSite, if a volumeVolume of a remote siteSite is replicated on local siteSite.
